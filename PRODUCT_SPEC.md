# 個體用戶 LLM 個股理財系統產品規格書

## 1. 文件目的

本文件將「個體用戶 LLM 個股理財系統」需求整理為可執行的產品與技術規格，作為產品、設計、後端、前端、資料工程與 LLM 工程的共同依據。

本系統目標是打造一個以大型語言模型（LLM）為核心、以公開資訊為基礎的個股理財助理，協助使用者快速理解單一股票的近期動態、財報重點、已知事實、可能影響與風險提醒，並盡可能降低幻覺與錯誤引用風險。

## 2. 產品定位

### 2.1 核心價值

- 降低使用者查找財經資訊的時間成本
- 從海量公開資料中萃取對投資判斷最有用的重點
- 以來源治理與交叉驗證降低錯誤資訊誤導
- 以白話、可驗證、可回溯的方式提升金融訊息可理解度

### 2.2 產品原則

- 簡潔：先結論、後細節，避免冗長輸出
- 易懂：使用白話描述，保留必要金融術語
- 可驗證：每一個核心結論都能對回原始資料
- 低幻覺：資料不足時明確拒答，不憑空補完
- 中立：不使用煽動性或情緒化投資語氣

### 2.3 目標用戶

- 對單一股票有明確關注標的的個人投資者
- 沒有時間長時間閱讀新聞、法說會與公告的用戶
- 希望快速理解「最近發生什麼」與「有哪些風險」的使用者

### 2.4 非目標範圍

- 不提供個人化投資建議、買賣點位或保證報酬
- 不做自動下單、不做投顧服務、不做資產配置推薦
- 不處理未公開資訊、內線資訊或私人社群消息

## 3. MVP 定義

### 3.1 MVP 功能範圍

- 支援單一股票查詢
- 支援三類主題查詢：新聞、財報、公告
- 支援時間範圍限定：近 1 天、近 7 天、近 30 天、最新一季
- 支援以 FinMind 為主資料來源的檢索與整理
- 支援摘要、事實整理、可能影響、風險提醒、來源清單輸出
- 支援信任度紅綠燈與原文對照

### 3.2 MVP 不含

- 多股票比較
- 技術分析圖表
- 即時盤中推播
- 使用者投資組合管理
- 自動化投資建議分數

## 4. 產品使用情境

### 4.1 主要查詢案例

- 「台積電最近 7 天有什麼重要新聞？」
- 「2330 最新一季財報重點是什麼？」
- 「這家公司最近可以買嗎？」
- 「某公司法說會講了什麼？有哪些風險？」

### 4.2 預期回應特色

- 先用一句話總結
- 條列 3 個最重要重點
- 明確標示哪些是已知事實、哪些是推論
- 補足反方觀點與風險提醒
- 顯示資料是否充足、是否有矛盾、是否過時

## 5. 整體系統架構

系統採 Six-Layer Architecture，讓輸入、檢索、治理、生成、驗證與呈現彼此解耦，降低錯誤傳遞風險。

### 5.1 Input Layer

職責：

- 辨識使用者意圖
- 萃取股票代號、公司名稱、查詢主題、時間範圍、問題方向
- 將自然語言轉換為結構化查詢條件

輸入欄位：

- `ticker`
- `company_name`
- `topic`
- `time_range`
- `question_type`
- `stance_bias`

範例：

- 使用者輸入：「為什麼現在可以買 2330？」
- 結構化結果：
  - `ticker=2330`
  - `topic=綜合`
  - `time_range=近30天`
  - `question_type=投資判斷輔助`
  - `stance_bias=偏多`

### 5.2 Retrieval Layer

職責：

- 透過 FinMind API 抓取公開資料
- 做關鍵字查詢與語意向量檢索
- 將資料組成 LLM 可引用的證據集合

檢索策略：

- 先做結構化篩選：股票代號、日期、主題、來源類型
- 再做向量召回：找出與問題語意最相關的內容片段
- 最後做 rerank：優先保留高可信、較新、與問題最接近的內容

### 5.3 Data Governance Layer

職責：

- 清洗、去重、正規化
- 驗證日期、標題、來源與內容完整性
- 給每筆資料可信度等級與時效性標記

此層為降低幻覺的核心模組。

### 5.4 Generation Layer

職責：

- 根據整理後證據集合產生摘要
- 嚴格遵守系統 Prompt 與格式約束
- 區分已知事實、合理推論與資料不足

LLM 角色：

- 財經資訊整理與風險控管助理

### 5.5 Validation Layer

職責：

- 對生成內容做一致性檢查
- 驗證是否引用到足夠證據
- 檢查是否違反「只能依資料回答」原則
- 在低信心時阻擋回應或降級輸出

### 5.6 Presentation Layer

職責：

- 將結果格式化為易讀 UI
- 顯示信任度燈號
- 提供摘要與原文對照
- 提供證據熱度與來源列表

## 6. 端到端流程

1. 使用者輸入自然語言問題
2. Input Layer 解析股票、主題、時間與問題方向
3. Retrieval Layer 向資料庫與 FinMind API 查詢候選資料
4. Data Governance Layer 做清洗、去重、可信度與時效判定
5. Generation Layer 根據受控 Prompt 生成答案草稿
6. Validation Layer 檢查答案是否有矛盾、缺證據或過度推論
7. Presentation Layer 產出最終回應與證據對照畫面

## 7. 資料來源策略

### 7.1 資料來源優先級

- 高可信：官方公告、公司官網、法說會逐字稿、交易所公開揭露資料
- 中可信：主流財經媒體
- 低可信：論壇、社群、內容農場、未具名轉述

MVP 建議：

- 只納入高可信與中可信資料
- 低可信來源不進入預設答案，可作為觀察資料但不可直接支撐結論

### 7.2 來源欄位

每筆資料至少需保存以下欄位：

- `source_id`
- `source_name`
- `source_type`
- `source_tier`
- `publisher`
- `published_at`
- `title`
- `url`
- `author`
- `content_raw`
- `content_clean`
- `ticker_tags`
- `topic_tags`

## 8. PostgreSQL + pgvector 設計

### 8.1 核心資料表

#### `documents`

儲存原始與清洗後資料。

建議欄位：

- `id` UUID PK
- `ticker` VARCHAR(16)
- `title` TEXT
- `summary_raw` TEXT
- `content_raw` TEXT
- `content_clean` TEXT
- `source_name` TEXT
- `source_tier` SMALLINT
- `source_type` TEXT
- `url` TEXT UNIQUE
- `author` TEXT
- `published_at` TIMESTAMP
- `fetched_at` TIMESTAMP
- `topic_tags` TEXT[]
- `is_valid` BOOLEAN
- `is_superseded` BOOLEAN DEFAULT FALSE
- `dedupe_group_id` UUID NULL
- `created_at` TIMESTAMP
- `updated_at` TIMESTAMP

#### `document_embeddings`

儲存文件或片段向量。

- `id` UUID PK
- `document_id` UUID FK
- `chunk_index` INT
- `chunk_text` TEXT
- `embedding` VECTOR(1536)
- `created_at` TIMESTAMP

#### `query_logs`

記錄使用者查詢與系統回應，用於調校與稽核。

- `id` UUID PK
- `user_query` TEXT
- `ticker`
- `topic`
- `time_range`
- `retrieved_doc_count` INT
- `confidence_score` NUMERIC(5,4)
- `validation_status` TEXT
- `response_text` TEXT
- `created_at` TIMESTAMP

### 8.2 索引建議

- `documents(ticker, published_at DESC)`
- `documents(source_tier, published_at DESC)`
- `documents(topic_tags)` GIN
- `document_embeddings using ivfflat (embedding vector_cosine_ops)`

### 8.3 混合檢索策略

查詢流程：

1. 先用 `ticker + time_range + source_tier` 做結構化篩選
2. 再用 `pgvector` 做語意相似度召回
3. 對結果加權排序：
   - 語意相關性
   - 來源等級
   - 新鮮度
   - 是否被多來源交叉驗證

建議排序分數：

`final_score = 0.45 * semantic_score + 0.25 * source_score + 0.20 * freshness_score + 0.10 * corroboration_score`

## 9. 去重與資料治理規則

### 9.1 向量化去重

目的：

- 避免同一新聞被多次轉載造成重複權重
- 保留最完整、最早或最高可信版本

規則建議：

- 先以 `url` 完全相同判定重複
- 再以 `title normalized` 做文字相似判斷
- 最後以 `content embedding cosine similarity >= 0.93` 視為高度重複

重複處理：

- 優先保留高來源等級版本
- 同來源時優先保留內容較完整版本
- 保留被淘汰文件與主文件映射，供追溯使用

### 9.2 資料品質控制表

| 項目 | 實作規則 |
| --- | --- |
| 來源分級 | 官方與交易所為高可信；主流媒體為中可信；論壇與農場為低可信 |
| 時效性控制 | 即時新聞優先看 1-7 天；財報優先最新一季；舊公告若已被新公告覆蓋則降權 |
| 交叉驗證 | 至少兩個高/中可信來源提及同事件可提高權重；單一來源需標註待確認 |
| 異常攔截 | 排除無日期、無 URL、無正文、純評論、標題與內文不符內容 |

### 9.3 資料狀態標籤

- `sufficient`
- `insufficient`
- `conflicting`
- `outdated`
- `pending_confirmation`

## 10. LLM Prompt 工程規範

### 10.1 System Prompt 核心規則

- 角色是「財經資訊整理與風險控管助理」
- 僅能根據檢索提供的資料回答
- 不可補充未出現在證據中的事實
- 若資料不足，必須明確說「資料不足，無法確認」
- 若資料互相矛盾，必須明確指出矛盾點
- 若問題帶有偏多或偏空傾向，必須補充至少 3 個反向觀點或風險點
- 嚴格區分「已知事實」與「可能影響」
- 不得提供投資建議語句，如「建議買進」「一定會漲」

### 10.2 建議提示結構

輸入給模型的內容建議包含：

- 使用者問題
- 結構化查詢條件
- 經治理後的證據摘要
- 可引用來源清單
- 輸出格式模板
- 驗證規則

### 10.3 輸出格式

1. 一句總結
2. 三大重點
3. 已知事實
4. 可能影響
5. 風險提醒
6. 資料狀態
7. 來源列表

### 10.4 回答風格範例

- 一句總結：台積電近 7 日資訊偏中性偏多，但仍需留意需求波動與外部產業風險。
- 已知事實：公司發布法說會內容、某媒體報導營收動能、最新公告揭露資本支出。
- 可能影響：若資本支出提高且需求延續，可能支撐中長期成長。
- 風險提醒：若終端需求不如預期、毛利率承壓或市場預期過高，股價可能波動。

## 11. Validation Layer 規格

### 11.1 Confidence Gate

系統需為每次回答計算 `confidence_score`，作為是否放行完整回覆的門檻。

建議分數來源：

- 證據數量
- 高可信來源占比
- 資料新鮮度
- 是否跨來源一致
- 是否出現互相矛盾資訊

建議門檻：

- `>= 0.80`：完整回答，標示綠燈
- `0.55 - 0.79`：回答但加註保留，標示黃燈
- `< 0.55`：降級回答，只輸出有限資訊與資料不足提示，標示紅燈

### 11.2 答案檢查規則

- 每一個結論都必須對應至少一筆證據
- 若引用的是單一來源，需顯示「待確認」
- 若答案含明確數字，需檢查數字是否存在於原始資料
- 若模型輸出與證據不一致，則直接攔截或重試
- 若風險提醒不足 3 點且問題具明顯單邊立場，則重生

### 11.3 降級策略

當驗證未通過時，依序採取以下措施：

1. 重新摘要一次，縮小可用證據範圍
2. 若仍失敗，只輸出已知事實與來源
3. 若資料嚴重不足，直接回覆「資料不足，無法確認」

## 12. UX 與信任機制

### 12.1 資訊分層

預設只顯示：

- 一句總結
- 三大重點
- 信任度燈號

展開後顯示：

- 白話翻譯
- 已知事實
- 可能影響
- 風險提醒
- 原文對照
- 來源列表

### 12.2 信任度紅綠燈

- 綠燈：來源以官方為主、資料完整、來源一致
- 黃燈：主流媒體為主、資訊局部不完整或有輕微矛盾
- 紅燈：來源不足、資訊衝突大或資料過舊

### 12.3 Grounding UX

系統需支援：

- 原文對照：AI 摘要可對應原文片段
- 證據熱度：同一結論被多少來源支持
- 來源類型標記：官方、媒體、其他
- 時間標記：每則證據的發布日期

## 13. API 與服務切分建議

### 13.1 服務模組

- `ingestion-service`：抓取 FinMind 與資料清洗
- `retrieval-service`：負責混合檢索與 rerank
- `llm-service`：負責 Prompt 組裝與生成
- `validation-service`：負責答案驗證與信心評分
- `api-service`：提供前端查詢 API

### 13.2 API 建議

#### `POST /api/query`

用途：

- 提交使用者查詢並取得整理後答案

Request：

```json
{
  "query": "2330 最近 7 天有什麼重點？",
  "ticker": "2330",
  "topic": "news",
  "timeRange": "7d"
}
```

Response：

```json
{
  "summary": "近 7 天資訊偏中性偏多，但仍有需求與估值風險。",
  "highlights": [
    "近期市場關注法說與營收動能",
    "高可信來源資訊一致度中高",
    "短期風險集中在外部需求變化"
  ],
  "facts": [],
  "impacts": [],
  "risks": [],
  "dataStatus": {
    "sufficiency": "sufficient",
    "consistency": "mostly_consistent",
    "freshness": "recent"
  },
  "confidenceLight": "yellow",
  "confidenceScore": 0.74,
  "sources": []
}
```

#### `GET /api/sources/:queryId`

用途：

- 取得該次回覆的原始證據與對照片段

## 14. 非功能需求

### 14.1 效能

- 一般查詢在 5-10 秒內完成
- 檢索層需支援熱門股票快取
- 對最新新聞可採短期快取與背景刷新

### 14.2 可觀測性

需記錄：

- 查詢延遲
- 各層耗時
- 檢索文件數
- 驗證失敗率
- 低信心回答比例

### 14.3 安全性

- 不儲存敏感個資
- API Key 與資料來源憑證應使用環境變數管理
- 回應需過濾 Prompt Injection 類型內容

## 15. 合規與風險控制

### 15.1 必備揭露

每次回覆結尾都必須附上：

「本系統僅整理公開資訊，不構成投資建議。」

### 15.2 語氣規範

- 客觀
- 中立
- 不承諾報酬
- 不下定論式鼓動買賣

### 15.3 風險項目

- 模型誤引資料
- 舊聞誤當新訊
- 單一來源造成偏誤
- 標題黨或評論文章誤導
- 使用者將資訊整理誤認為投資建議

## 16. 驗收標準

### 16.1 功能驗收

- 能正確解析股票代號、主題與時間範圍
- 能從資料庫召回與問題相關的高可信內容
- 能去除明顯重複新聞
- 能在資料不足時明確拒答
- 能在偏單邊問題中補足至少 3 點反向風險
- 能顯示來源列表與原文對照

### 16.2 品質驗收

- 同一查詢結果具有高度穩定性
- 重要結論皆可追溯到證據
- 回答不得出現未在資料中出現的具體事實
- 過時資料不應高於新資料排序

## 17. 開發里程碑建議

### Phase 1：資料底座

- 建立 FinMind ingestion pipeline
- 建立 PostgreSQL 與 pgvector schema
- 完成去重、清洗、來源分級

### Phase 2：查詢閉環

- 完成 Input、Retrieval、Generation 基本流程
- 串接 LLM 與固定輸出模板
- 建立 query log

### Phase 3：低幻覺強化

- 補上 Validation Layer
- 加入 confidence gate
- 完成原文對照與證據熱度

### Phase 4：產品化

- 建立前端 UI
- 優化快取、延遲與觀測
- 補齊合規揭露與操作紀錄
