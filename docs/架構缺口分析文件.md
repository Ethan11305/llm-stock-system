## 缺口零（P0-最高優先）：Embedding 向量寫入與語意檢索

### 現狀問題

系統的 RAG 架構中，「R」（Retrieval）目前**完全沒有語意檢索能力**。

**資料庫層面：基礎設施已就緒但從未使用**

`db/sql/002_schema.sql` 中已建立：
- `document_embeddings` 表：含 `embedding VECTOR(1536)` 欄位
- IVFFlat 索引：`idx_document_embeddings_ivfflat USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100)`
- pgvector 擴展已安裝（`001_extensions.sql`）

但整個 `src/` 中沒有任何程式碼會生成 embedding、寫入此表、或查詢此表。

**檢索層面：程式化合成文件，非語意搜尋**

目前的檢索流程：
```
StructuredQuery
  → RetrievalLayer.retrieve(query)
    → PostgresMarketDocumentRepository.search_documents(query)
      → FinMindPostgresGateway.build_documents(query)
        → _resolve_retrieval_profile(query)  # 根據 intent/question_type 選 profile
        → 依序呼叫 profile.builder_plan 中的 builder methods
          例：_build_price_documents(), _build_news_documents(), ...
        → _sorted(documents, profile)  # 按 sort_priority → tier → date 排序
  → documents[:max_documents]  # 截取前 N 篇
```

這代表所有文件都是**從結構化資料表（daily_price_bars、financial_statement_items 等）程式化組合而成**的，不是從文件庫中根據使用者查詢語意去搜尋。

**新聞去重也不是用 embedding**

`news_pipeline.py` 中的去重是 **title + URL 完全比對**，沒有使用 embedding cosine similarity。

### 這導致什麼具體問題？

1. **無法處理模糊語意查詢**：使用者問「台積電最近有什麼供應鏈風險？」時，系統只能靠 metadata（ticker=2330, facet=NEWS）把所有新聞都拉出來，無法精準找到跟「供應鏈風險」語意相關的段落
2. **Context 窗口浪費**：把所有新聞塞給 LLM，大量不相關內容佔用 token，降低生成品質
3. **無法跨文件關聯**：同一個事件（如「美國出口管制」）可能出現在新聞、法說會紀要、產業報告中，但目前無法透過語意相似度把它們關聯起來
4. **Retrieval Profile 維護成本高**：每新增一個查詢場景就要寫新的 builder + profile，而語意檢索可以自動適應新場景

### 方案：完整的 Embedding Pipeline + Hybrid Retrieval

整體架構分為三個階段：**寫入（Ingestion）→ 儲存（Storage）→ 檢索（Retrieval）**

#### 階段 1：Embedding Model 選擇

| 模型 | 維度 | 中文能力 | 延遲 | 成本 | 建議 |
|---|---|---|---|---|---|
| OpenAI `text-embedding-3-small` | 1536 | 良好 | ~100ms/batch | $0.02/1M tokens | ✅ 首選（與現有 schema VECTOR(1536) 一致） |
| OpenAI `text-embedding-3-large` | 3072 | 優秀 | ~150ms/batch | $0.13/1M tokens | 需改 schema |
| `intfloat/multilingual-e5-large` | 1024 | 優秀 | 本地推論 | 免費 | 需 GPU，需改 schema |

**建議：** 使用 `text-embedding-3-small`，因為你的 schema 已經是 `VECTOR(1536)`，零改動。

#### 階段 2：Document Chunking 策略

**新增檔案：** `src/llm_stock_system/services/document_chunker.py`

```python
from dataclasses import dataclass

@dataclass
class DocumentChunk:
    """文件切片，作為 embedding 的最小單位。"""
    document_id: str
    chunk_index: int
    chunk_text: str
    metadata: dict  # ticker, source_tier, published_at 等，用於後續過濾

class DocumentChunker:
    """將 Document 切成適合 embedding 的 chunk。

    切片策略依文件類型而異：
    - 新聞文章：按段落切，每段 200-500 tokens
    - 財報數據：整份作為一個 chunk（已經是結構化摘要）
    - 法說會逐字稿：按發言段落切，保留講者資訊
    """

    def __init__(
        self,
        max_chunk_tokens: int = 500,
        overlap_tokens: int = 50,
        min_chunk_tokens: int = 30,
    ):
        self.max_chunk_tokens = max_chunk_tokens
        self.overlap_tokens = overlap_tokens
        self.min_chunk_tokens = min_chunk_tokens

    def chunk_document(self, document: Document) -> list[DocumentChunk]:
        """將單一 Document 切成多個 chunk。"""

        # 短文件（< max_chunk_tokens）：整篇作為一個 chunk
        content = f"{document.title}\n\n{document.content}"
        estimated_tokens = len(content) // 2  # 中文約 1 字 ≈ 0.5 token（粗估）

        if estimated_tokens <= self.max_chunk_tokens:
            return [DocumentChunk(
                document_id=document.id,
                chunk_index=0,
                chunk_text=content,
                metadata=self._extract_metadata(document),
            )]

        # 長文件：按段落切，帶重疊
        paragraphs = content.split("\n")
        chunks = []
        current_text = ""
        chunk_index = 0

        for para in paragraphs:
            para = para.strip()
            if not para:
                continue

            candidate = f"{current_text}\n{para}" if current_text else para
            candidate_tokens = len(candidate) // 2

            if candidate_tokens > self.max_chunk_tokens and current_text:
                # 儲存當前 chunk
                chunks.append(DocumentChunk(
                    document_id=document.id,
                    chunk_index=chunk_index,
                    chunk_text=current_text.strip(),
                    metadata=self._extract_metadata(document),
                ))
                chunk_index += 1

                # 帶重疊：取 current_text 最後 overlap_tokens 的部分
                overlap_chars = self.overlap_tokens * 2
                overlap_text = current_text[-overlap_chars:] if len(current_text) > overlap_chars else ""
                current_text = f"{overlap_text}\n{para}" if overlap_text else para
            else:
                current_text = candidate

        # 最後一段
        if current_text and len(current_text) // 2 >= self.min_chunk_tokens:
            chunks.append(DocumentChunk(
                document_id=document.id,
                chunk_index=chunk_index,
                chunk_text=current_text.strip(),
                metadata=self._extract_metadata(document),
            ))

        return chunks

    def _extract_metadata(self, document: Document) -> dict:
        return {
            "ticker": document.ticker,
            "source_tier": document.source_tier.value,
            "source_name": document.source_name,
            "published_at": document.published_at.isoformat(),
            "topics": [t.value for t in document.topics],
        }
```

#### 階段 3：Embedding 生成與寫入

**新增檔案：** `src/llm_stock_system/services/embedding_service.py`

```python
import hashlib
from dataclasses import dataclass

@dataclass
class EmbeddingResult:
    chunk: DocumentChunk
    vector: list[float]  # 1536 維

class EmbeddingService:
    """負責生成 embedding 並寫入 document_embeddings 表。

    設計要點：
    - 批次處理（OpenAI 一次最多 2048 個輸入）
    - 重複偵測（避免同一 chunk 重複 embed）
    - 錯誤重試（API 可能 rate limit）
    """

    def __init__(
        self,
        openai_api_key: str,
        model: str = "text-embedding-3-small",
        batch_size: int = 100,
        db_engine = None,  # SQLAlchemy engine
    ):
        self.api_key = openai_api_key
        self.model = model
        self.batch_size = batch_size
        self.engine = db_engine
        self._session = None

    # ──────────────────────────────────────
    # 公開 API
    # ──────────────────────────────────────

    def embed_and_store(self, documents: list[Document], chunker: DocumentChunker) -> int:
        """完整流程：Document → Chunk → Embed → Store。回傳寫入筆數。"""

        # Step 1: Chunking
        all_chunks: list[DocumentChunk] = []
        for doc in documents:
            all_chunks.extend(chunker.chunk_document(doc))

        if not all_chunks:
            return 0

        # Step 2: 過濾已存在的 chunk（避免重複寫入）
        new_chunks = self._filter_existing(all_chunks)

        if not new_chunks:
            return 0

        # Step 3: 批次生成 embedding
        results: list[EmbeddingResult] = []
        for i in range(0, len(new_chunks), self.batch_size):
            batch = new_chunks[i : i + self.batch_size]
            vectors = self._call_openai_embedding([c.chunk_text for c in batch])
            for chunk, vector in zip(batch, vectors):
                results.append(EmbeddingResult(chunk=chunk, vector=vector))

        # Step 4: 批次寫入 DB
        written = self._bulk_insert(results)
        return written

    # ──────────────────────────────────────
    # OpenAI API 呼叫
    # ──────────────────────────────────────

    def _call_openai_embedding(self, texts: list[str]) -> list[list[float]]:
        """呼叫 OpenAI Embedding API，帶重試。"""
        import httpx
        import time

        url = "https://api.openai.com/v1/embeddings"
        headers = {
            "Authorization": f"Bearer {self.api_key}",
            "Content-Type": "application/json",
        }
        payload = {
            "model": self.model,
            "input": texts,
        }

        for attempt in range(3):
            try:
                resp = httpx.post(url, json=payload, headers=headers, timeout=30)
                resp.raise_for_status()
                data = resp.json()
                # OpenAI 回傳的 embedding 順序與 input 一致
                return [item["embedding"] for item in sorted(data["data"], key=lambda x: x["index"])]
            except httpx.HTTPStatusError as e:
                if e.response.status_code == 429:  # Rate limit
                    time.sleep(2 ** attempt)
                    continue
                raise
        raise RuntimeError("OpenAI Embedding API 重試 3 次仍失敗")

    # ──────────────────────────────────────
    # DB 操作
    # ──────────────────────────────────────

    def _filter_existing(self, chunks: list[DocumentChunk]) -> list[DocumentChunk]:
        """過濾已經在 DB 中的 chunk，避免重複寫入。
        用 (document_id, chunk_index) 作為唯一鍵。
        """
        from sqlalchemy import text

        doc_ids = list({c.document_id for c in chunks})

        with self.engine.connect() as conn:
            result = conn.execute(
                text("SELECT document_id, chunk_index FROM document_embeddings WHERE document_id = ANY(:ids)"),
                {"ids": doc_ids},
            )
            existing = {(str(row[0]), row[1]) for row in result}

        return [c for c in chunks if (c.document_id, c.chunk_index) not in existing]

    def _bulk_insert(self, results: list[EmbeddingResult]) -> int:
        """批次寫入 document_embeddings 表。"""
        from sqlalchemy import text
        import uuid

        if not results:
            return 0

        sql = text("""
            INSERT INTO document_embeddings (id, document_id, chunk_index, chunk_text, embedding)
            VALUES (:id, :document_id, :chunk_index, :chunk_text, :embedding)
            ON CONFLICT (document_id, chunk_index) DO UPDATE
                SET chunk_text = EXCLUDED.chunk_text,
                    embedding = EXCLUDED.embedding
        """)

        rows = []
        for r in results:
            rows.append({
                "id": str(uuid.uuid4()),
                "document_id": r.chunk.document_id,
                "chunk_index": r.chunk.chunk_index,
                "chunk_text": r.chunk.chunk_text,
                "embedding": str(r.vector),  # pgvector 接受 '[0.1, 0.2, ...]' 格式
            })

        with self.engine.begin() as conn:
            conn.execute(sql, rows)

        return len(rows)
```

**注意：** 上面的 `ON CONFLICT (document_id, chunk_index)` 需要在 schema 上加一個 unique constraint：

```sql
-- db/sql/004_embedding_unique_constraint.sql

    ADD CONSTRAINT uq_document_embeddings_doc_chunk
    UNIQUE (document_id, chunk_index);
```

#### 階段 4：語意檢索查詢

**新增檔案：** `src/llm_stock_system/adapters/vector_retrieval.py`

```python
from dataclasses import dataclass

@dataclass
class SemanticSearchResult:
    document_id: str
    chunk_text: str
    chunk_index: int
    similarity_score: float  # 1 - cosine distance（越高越相似）
    # 從 documents 表 JOIN 來的 metadata
    ticker: str | None = None
    title: str | None = None
    source_tier: str | None = None
    published_at: str | None = None

class VectorRetrievalAdapter:
    """基於 pgvector 的語意檢索。

    核心查詢：
    1. 將使用者查詢轉成 embedding
    2. 用 cosine similarity 在 document_embeddings 中找最相似的 chunk
    3. JOIN documents 表取得完整 metadata
    4. 支援 metadata 前置過濾（ticker、時間範圍、source_tier）
    """

    def __init__(self, embedding_service: EmbeddingService, db_engine):
        self.embedding_service = embedding_service
        self.engine = db_engine

    def search(
        self,
        query_text: str,
        top_k: int = 10,
        ticker: str | None = None,
        source_tier_min: str | None = None,  # "high", "medium", "low"
        days_back: int | None = None,
        similarity_threshold: float = 0.65,
    ) -> list[SemanticSearchResult]:
        """語意檢索主方法。"""
        from sqlalchemy import text

        # Step 1: 查詢文本 → embedding
        query_vector = self.embedding_service._call_openai_embedding([query_text])[0]
        vector_str = str(query_vector)

        # Step 2: 建構 SQL（pgvector cosine distance: <=> 運算子）
        #   1 - (embedding <=> query_vector) = cosine similarity
        where_clauses = []
        params = {"query_vec": vector_str, "top_k": top_k, "threshold": similarity_threshold}

        if ticker:
            where_clauses.append("d.ticker = :ticker")
            params["ticker"] = ticker

        if source_tier_min:
            tier_order = {"high": 3, "medium": 2, "low": 1}
            min_order = tier_order.get(source_tier_min, 1)
            # 只取 >= 指定層級的來源
            tier_values = [k for k, v in tier_order.items() if v >= min_order]
            where_clauses.append("d.source_tier = ANY(:tier_values)")
            params["tier_values"] = tier_values

        if days_back:
            where_clauses.append("d.published_at >= CURRENT_DATE - :days_back * INTERVAL '1 day'")
            params["days_back"] = days_back

        where_sql = f"WHERE {' AND '.join(where_clauses)}" if where_clauses else ""

        sql = text(f"""
            SELECT
                de.document_id,
                de.chunk_text,
                de.chunk_index,
                1 - (de.embedding <=> :query_vec::vector) AS similarity,
                d.ticker,
                d.title,
                d.source_tier,
                d.published_at
            FROM document_embeddings de
            JOIN documents d ON d.id = de.document_id
            {where_sql}
                {"AND" if where_clauses else "WHERE"}
                1 - (de.embedding <=> :query_vec::vector) >= :threshold
            ORDER BY de.embedding <=> :query_vec::vector ASC
            LIMIT :top_k
        """)

        with self.engine.connect() as conn:
            rows = conn.execute(sql, params).fetchall()

        return [
            SemanticSearchResult(
                document_id=str(row[0]),
                chunk_text=row[1],
                chunk_index=row[2],
                similarity_score=float(row[3]),
                ticker=row[4],
                title=row[5],
                source_tier=row[6],
                published_at=str(row[7]) if row[7] else None,
            )
            for row in rows
        ]
```

#### 階段 5：Hybrid Retrieval（最關鍵的整合）

**改動檔案：** `src/llm_stock_system/layers/retrieval_layer.py`

目前的 `RetrievalLayer` 只有 5 行程式碼，只做 `repository.search_documents(query)[:max_documents]`。改造為 Hybrid 模式：

```python
class HybridRetrievalLayer:
    """混合檢索層：結合現有的 metadata 檢索 + 新的語意檢索。

    策略：
    1. 兩路並行：metadata 檢索（現有 builder 流程）+ 語意檢索（新 vector 流程）
    2. 結果合併去重（同一 document_id 只保留一份）
    3. 混合排序：語意分數 × 0.6 + metadata 排序分數 × 0.4
    """

    def __init__(
        self,
        document_repository: DocumentRepository,       # 現有的 metadata 檢索
        vector_adapter: VectorRetrievalAdapter | None,  # 新的語意檢索（可選）
        max_documents: int = 8,
        semantic_weight: float = 0.6,
        metadata_weight: float = 0.4,
    ):
        self._document_repository = document_repository
        self._vector_adapter = vector_adapter
        self._max_documents = max_documents
        self._semantic_weight = semantic_weight
        self._metadata_weight = metadata_weight

    def retrieve(self, query: StructuredQuery) -> list[Document]:
        """混合檢索。如果 vector_adapter 不可用，graceful fallback 到純 metadata。"""

        # Path A: 現有 metadata 檢索（保證永遠有結果）
        metadata_docs = self._document_repository.search_documents(query)

        # Path B: 語意檢索（可選）
        if self._vector_adapter is None:
            return metadata_docs[: self._max_documents]

        try:
            semantic_results = self._vector_adapter.search(
                query_text=query.user_query,
                top_k=self._max_documents * 2,  # 多拉一些，後面合併時去重
                ticker=query.ticker,
                days_back=query.time_range_days,
            )
        except Exception:
            # 語意檢索失敗時 graceful fallback
            return metadata_docs[: self._max_documents]

        # 合併兩路結果
        return self._merge_results(metadata_docs, semantic_results)

    def _merge_results(
        self,
        metadata_docs: list[Document],
        semantic_results: list[SemanticSearchResult],
    ) -> list[Document]:
        """合併兩路結果，混合排序。"""

        # 建立 document_id → semantic_score 的映射
        semantic_scores: dict[str, float] = {
            r.document_id: r.similarity_score for r in semantic_results
        }

        # 為 metadata_docs 計算正規化排序分數（排名越前分數越高）
        metadata_scores: dict[str, float] = {}
        for i, doc in enumerate(metadata_docs):
            # 正規化到 0-1：第 1 名 = 1.0，最後一名 ≈ 0.0
            metadata_scores[doc.id] = max(0, 1.0 - i / max(len(metadata_docs), 1))

        # 合併所有唯一 document_id
        all_doc_ids = set(metadata_scores.keys()) | set(semantic_scores.keys())

        # 建立 document_id → Document 的映射
        doc_map = {doc.id: doc for doc in metadata_docs}

        # 對於只在 semantic_results 中出現的文件，需要從 DB 補查完整 Document
        # （semantic_results 只有 chunk_text，沒有完整 Document）
        missing_ids = [did for did in semantic_scores if did not in doc_map]
        if missing_ids:
            supplementary_docs = self._fetch_documents_by_ids(missing_ids)
            doc_map.update({doc.id: doc for doc in supplementary_docs})

        # 混合分數計算
        scored: list[tuple[str, float]] = []
        for doc_id in all_doc_ids:
            if doc_id not in doc_map:
                continue
            sem_score = semantic_scores.get(doc_id, 0.0)
            meta_score = metadata_scores.get(doc_id, 0.0)
            hybrid_score = sem_score * self._semantic_weight + meta_score * self._metadata_weight
            scored.append((doc_id, hybrid_score))

        # 按混合分數降序排列
        scored.sort(key=lambda x: x[1], reverse=True)

        return [doc_map[doc_id] for doc_id, _ in scored[:self._max_documents]]

    def _fetch_documents_by_ids(self, doc_ids: list[str]) -> list[Document]:
        """從 documents 表補查完整 Document 物件。"""
```

#### 階段 6：Embedding 寫入觸發時機

**改動檔案：** `src/llm_stock_system/services/query_data_hydrator.py`

在現有的資料拉取流程中，每次 hydrate 成功後觸發 embedding 生成：

```python
class QueryDataHydrator:
    def __init__(self, ..., embedding_service: EmbeddingService | None = None, chunker: DocumentChunker | None = None):
        self._embedding_service = embedding_service
        self._chunker = chunker or DocumentChunker()

    def hydrate(self, query: StructuredQuery) -> HydrationResult:
        result = self._original_hydrate(query)  # 現有邏輯

        # 新增：對本次拉取到的新文件生成 embedding
        if self._embedding_service and result.new_documents:
            try:
                self._embedding_service.embed_and_store(
                    result.new_documents, self._chunker
                )
            except Exception as e:
```

```python
def backfill(db_engine, embedding_service, chunker, batch_size=100):
    """找出 documents 表中尚未有 embedding 的文件，批次處理。"""
    from sqlalchemy import text

    # 找出沒有對應 embedding 的 document_id
    sql = text("""
        SELECT d.id, d.ticker, d.title, d.content_clean, d.source_name,
               d.source_type, d.source_tier, d.url, d.published_at
        FROM documents d
        LEFT JOIN document_embeddings de ON de.document_id = d.id
        WHERE de.id IS NULL AND d.is_valid = TRUE
        ORDER BY d.published_at DESC
        LIMIT :batch_size
    """)

    with db_engine.connect() as conn:
        rows = conn.execute(sql, {"batch_size": batch_size}).fetchall()

    if not rows:
        print("所有文件都已經有 embedding，無需回填。")
        return 0

    documents = [row_to_document(row) for row in rows]
    written = embedding_service.embed_and_store(documents, chunker)
    print(f"本批次回填 {written} 個 chunk 的 embedding（來自 {len(documents)} 篇文件）。")
    return written
```

## 缺口一：並行處理與串流回應 (Parallel Processing & Streaming)

### 現狀問題

`QueryDataHydrator.hydrate()` 中，9 種資料面向（PRICE_HISTORY、NEWS、FINANCIAL_STATEMENTS 等）是**依序同步拉取**的：

```python
# query_data_hydrator.py 現行邏輯（簡化）
for ticker in tickers:
    for facet in self._ordered_facets():
        result = self._sync_facet(ticker, facet, ...)  # 阻塞等待
```

假設每個 facet 平均耗時 1.5 秒，9 個 facet 就是 ~13.5 秒。加上 LLM 生成 ~3 秒，使用者需等待 16+ 秒才能看到任何回應。

### 方案 A：Facet 並行拉取（最小改動，建議優先實作）

**改動檔案：** `src/llm_stock_system/services/query_data_hydrator.py`

**核心思路：** 將 `_sync_facet()` 改為 async，用 `asyncio.gather()` 並行拉取同一 ticker 的所有 facet。

```python
import asyncio
from concurrent.futures import ThreadPoolExecutor

class QueryDataHydrator:
    def __init__(self, ...):
        self._executor = ThreadPoolExecutor(max_workers=6)

    async def hydrate_async(self, query: StructuredQuery) -> HydrationResult:
        """並行版本的 hydrate，取代原本的同步逐一拉取。"""
        if not query.tickers:
            return HydrationResult.empty()

        tasks = []
        for ticker in query.tickers:
            for facet in self._ordered_facets():
                tasks.append(self._sync_facet_async(ticker, facet, ...))

        results = await asyncio.gather(*tasks, return_exceptions=True)
        return self._aggregate_results(results)

    async def _sync_facet_async(self, ticker, facet, ...):
        """將原本阻塞的 _sync_facet 包裝成 async。
        如果底層 gateway 是同步 I/O（如 requests），用 executor 包裝。
        """
        loop = asyncio.get_event_loop()
        return await loop.run_in_executor(
            self._executor,
            self._sync_facet, ticker, facet, ...
        )
```

**Pipeline 層面的配合改動：** `src/llm_stock_system/orchestrator/pipeline.py`

> **⚠️ 重要：Hydration 必須在 Retrieval 之前完成。** Retrieval 需要讀取 hydration 剛同步回來的資料（新聞、股價等），如果兩者並行跑，retrieval 會讀到舊資料或空資料。正確的順序是：**facet 內部並行 → hydration 完成 → 再 retrieve。**

```python
class QueryPipeline:
    async def handle_query_async(self, request: QueryRequest) -> QueryResponse:
        query = self.input_layer.parse(request)

        # Step 1: Hydration（facet 內部並行，但整體必須先完成）
        hydration_result = await self.hydrator.hydrate_async(query)

        # Step 2: Retrieval（吃到剛同步回來的最新資料）
        documents = await asyncio.to_thread(self.retrieval_layer.retrieve, query)

        # Step 3: 後續步驟保持不變
        report = self.governance_layer.curate(query, documents)
        draft = self.generation_layer.generate(query, report)
        result = self.validation_layer.validate(query, report, draft, ...)
        return self.presentation_layer.present(draft, report, result)
```

**預估效果：** 9 個 facet 並行 → hydration 延遲從 ~13.5s 降至 ~2-3s（取決於最慢的 facet）。整體 pipeline 省下 ~10s。

### 方案 B：SSE 串流回應（進階，提升感知速度）

**改動檔案：** FastAPI 路由層 + `PresentationLayer`

**核心思路：** 不等所有層跑完才回應，改用 Server-Sent Events 分段推送。

```python
from fastapi.responses import StreamingResponse
import json

@app.post("/api/query/stream")
async def stream_query(request: QueryRequest):
    async def event_generator():
        query = input_layer.parse(request)

        # Phase 1：立即回傳解析結果，讓前端顯示「正在查詢 台積電 的新聞...」
        yield f"data: {json.dumps({'phase': 'parsing', 'ticker': query.tickers, 'intent': query.intent.value})}\n\n"

        # Phase 2：資料拉取（並行），每完成一個 facet 就推送進度
        async for facet_result in hydrator.hydrate_streaming(query):
            yield f"data: {json.dumps({'phase': 'hydrating', 'facet': facet_result.facet.value, 'status': facet_result.status})}\n\n"

        # Phase 3：生成答案
        yield f"data: {json.dumps({'phase': 'generating'})}\n\n"
        draft = generation_layer.generate(query, report)

        # Phase 4：驗證 + 最終結果
        result = validation_layer.validate(...)
        response = presentation_layer.present(draft, report, result)
        yield f"data: {json.dumps({'phase': 'complete', 'data': response.dict()})}\n\n"

    return StreamingResponse(event_generator(), media_type="text/event-stream")
```

**Hydrator 的 streaming 版本：**

```python
async def hydrate_streaming(self, query: StructuredQuery):
    """每完成一個 facet 就 yield 結果，而非等全部完成。"""
    tasks = {}
    for ticker in query.tickers:
        for facet in self._ordered_facets():
            task = asyncio.create_task(self._sync_facet_async(ticker, facet, ...))
            tasks[task] = facet

    for completed in asyncio.as_completed(tasks.keys()):
        result = await completed
        yield FacetStreamEvent(facet=tasks[completed], status="ok" if result.success else "fail")
```

## 缺口二：Routing / Policy Registry 重構

### 現狀問題

> **⚠️ 為什麼不是直接做 QueryController？** 目前 `InputLayer.parse()` (line 451) 和 `_detect_question_type()` (line 860) 已經是一套龐大的 routing brain，包含 27+ 種 question_type 的關鍵字規則。如果此時再加一個 QueryController 作為第二個 routing brain，會產生**兩套決策邏輯並存**的問題：InputLayer 決定「這是什麼類型的問題」，Controller 又決定「這個問題要拉哪些 facet」，兩者的判斷可能互相矛盾。
>
> 這也和 `responsibility-distribution.md` (line 52) 的判讀一致：**先拆 resolver / policy registry，比先做 agentic controller 更穩。**

**現狀的具體問題：**

1. `_detect_question_type()` 是一個 ~400 行的規則方法，混合了意圖辨識、facet 決定、topic tag 解析等多種職責
2. Intent → question_type → RetrievalProfile 的映射散落在 InputLayer 和 FinMindPostgresGateway 兩處
3. 新增一個查詢場景需要同時改 InputLayer 的 keyword 規則 + Gateway 的 RetrievalProfile + ValidationProfile

### 方案：拆分為 Policy Registry

**目標：** 把「什麼查詢需要什麼資料、什麼驗證規則」集中到一處，讓 InputLayer 只做解析，不做策略決定。

**新增檔案：** `src/llm_stock_system/core/query_policy.py`

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class QueryPolicy:
    """單一查詢策略：描述某種查詢意圖需要什麼資源與規則。

    將目前散落在 InputLayer、RetrievalProfile、ValidationProfile
    三處的配置集中到這裡。
    """
    intent: Intent
    question_type: str

    # 資料需求（目前在 INTENT_FACET_SPECS + RetrievalProfile 中）
    required_facets: frozenset[DataFacet]
    preferred_facets: frozenset[DataFacet]
    builder_plan: tuple[str, ...]          # 原 RetrievalProfile.builder_plan
    sort_priority: dict[str, int]          # 原 RetrievalProfile.sort_priority

    # 驗證需求（目前在 ValidationProfile 中）
    validation_rules: list                  # 原 ValidationProfile.rules
    confidence_cap: float | None = None     # 某些查詢類型的信心分上限

    # Hydration 配置（目前在 _ordered_facets 的固定順序中）
    facet_priority: list[DataFacet] | None = None  # 控制拉取順序

class PolicyRegistry:
    """集中管理所有 QueryPolicy。

    取代目前分散在三處的配置：
    - InputLayer._detect_question_type()  → 匹配邏輯
    - FinMindPostgresGateway._resolve_retrieval_profile() → 資料需求
    - ValidationProfile 字典 → 驗證規則
    """

    def __init__(self):
        self._policies: dict[tuple[Intent, str], QueryPolicy] = {}
        self._register_defaults()

    def resolve(self, intent: Intent, question_type: str) -> QueryPolicy:
        """根據 intent + question_type 查找對應的 policy。"""
        key = (intent, question_type)
        if key in self._policies:
            return self._policies[key]
        # Fallback: 只用 intent 匹配
        for (i, qt), policy in self._policies.items():
            if i == intent:
                return policy
        return self._default_policy()

    def _register_defaults(self):
        """將現有的散落配置遷移到這裡。

        遷移來源：
        - INTENT_FACET_SPECS (models.py)
        - _RETRIEVAL_PROFILES (postgres_market_data.py)
        - VALIDATION_PROFILES (validation_profiles.py)
        """
        # 範例：NEWS_DIGEST + 航運
        self._policies[(Intent.NEWS_DIGEST, "news_shipping")] = QueryPolicy(
            intent=Intent.NEWS_DIGEST,
            question_type="news_shipping",
            required_facets=frozenset({DataFacet.NEWS}),
            preferred_facets=frozenset({DataFacet.PRICE_HISTORY}),
            builder_plan=("_build_shipping_rate_impact_documents",),
            sort_priority={"shipping_rate": 1, "news": 2},
            validation_rules=[...],  # 從 validation_profiles.py 遷移
            facet_priority=[DataFacet.NEWS, DataFacet.PRICE_HISTORY],
        )
        # ... 其他 policies 依同模式遷移
```

### 遷移策略（漸進式，不要一次改完）

**Phase 1：建立 Registry，與現有邏輯並存**
- 新增 `PolicyRegistry`，先只註冊 3-5 個最常用的 policy
- Pipeline 中優先查 Registry，找不到就 fallback 到現有邏輯
- 這確保不會一次動太多程式碼

**Phase 2：逐步遷移現有規則**
- 每次 PR 遷移 3-5 個 question_type 的配置到 Registry
- 遷移完成後刪除 InputLayer 中對應的 if/elif 分支

**Phase 3：InputLayer 瘦身**
- InputLayer 只負責：ticker 辨識、公司名解析、intent 分類
- 不再做 question_type 到 facet/profile 的映射（由 Registry 負責）

### 為什麼先做這個，再考慮 QueryController

PolicyRegistry 完成後，如果未來要加 QueryController（Agentic 控制節點），它只需要查 Registry 就能知道「這個查詢需要什麼」，不會跟 InputLayer 的規則打架。Registry 是 Controller 的前置條件，不是替代品。

```
目前：InputLayer（routing brain #1）→ RetrievalProfile（routing brain #2）→ ValidationProfile（routing brain #3）
目標：InputLayer（純解析）→ PolicyRegistry（唯一的策略來源）→ Pipeline（按策略執行）
未來：InputLayer → PolicyRegistry → QueryController（可選的動態決策層）→ Pipeline
```

## 缺口三：Chain-of-Verification (CoVe) 自我驗證機制

> **前置條件：** P0（Embedding + 語意檢索）應先完成。CoVe 驗證的效果高度依賴 evidence 品質——如果 retrieval 拉回來的文件本身就不夠精準，驗證結果也不可靠。先提升 retrieval 品質，再做 CoVe 才有意義。

### 現狀問題

目前的驗證並非零分。OpenAI 路徑已有 grounded prompt（`openai_responses.py` line 100）強制模型只能用提供的 evidence 回答，preliminary fallback（line 226）處理無證據情況，`ValidationLayer`（line 22）也會對無證據回答降權。

**真正缺的是：** 即使有 grounded prompt，LLM 仍可能在數字上產生幻覺（例如把「營收成長 12%」寫成「營收成長 15%」）。目前的規則驗證無法抓到這類細微的數字錯誤。

### 啟用範圍限縮

> **⚠️ 不應全面啟用。** CoVe 每次多 2 次 LLM 呼叫，全面啟用會讓延遲和成本翻倍。

**僅在以下條件同時成立時啟用：**
1. 信心分落在 **YELLOW 區間**（0.55-0.79）—— GREEN 不需要，RED 資料本身就不足
2. 回答包含**具體數字**（營收、EPS、漲跌幅等）—— 純定性摘要幻覺風險低
3. 查詢意圖為 **EARNINGS_REVIEW、VALUATION_CHECK、FINANCIAL_HEALTH** —— 這三類最容易產生數字幻覺

```python
def should_enable_cove(validation_result, answer_draft, query) -> bool:
    """判斷是否啟用 CoVe。"""
    if validation_result.confidence_light != ConfidenceLight.YELLOW:
        return False
    if query.intent not in (Intent.EARNINGS_REVIEW, Intent.VALUATION_CHECK, Intent.FINANCIAL_HEALTH):
        return False
    # 檢查回答中是否包含數字
    import re
    has_numbers = bool(re.search(r'\d+\.?\d*%|\d{2,}', answer_draft.summary))
    return has_numbers
```

### 方案：兩階段 CoVe 驗證

**新增檔案：** `src/llm_stock_system/layers/cove_verifier.py`

```python
from dataclasses import dataclass

@dataclass
class VerificationClaim:
    """從 LLM 回答中提取的單一事實宣稱。"""
    claim_text: str                    # 例：「台積電 Q3 營收成長 12%」
    source_fragment: str | None = None # 對應的原文片段
    is_verified: bool = False
    verification_note: str = ""

@dataclass
class CoVeResult:
    claims: list[VerificationClaim]
    verified_ratio: float              # 已驗證 / 總宣稱數
    unverified_claims: list[str]       # 無法驗證的宣稱
    confidence_adjustment: float       # 建議的信心分調整值

class ChainOfVerificationLayer:
    """在 GenerationLayer 之後、ValidationLayer 之前執行。
    
    流程：
    1. 從 LLM 回答中提取事實宣稱（數字、日期、比較、因果關係）
    2. 將每個宣稱與原始證據做比對
    3. 標記無法驗證的宣稱
    4. 回饋給 ValidationLayer 調整信心分
    """

    CLAIM_EXTRACTION_PROMPT = """你是一個事實查核助理。請從以下回答中提取所有**可驗證的事實宣稱**。

回答內容：
{answer_text}

請列出每一個包含具體數字、日期、公司名稱、或因果關係的宣稱。
格式：每行一個宣稱，純文字，不要編號。

只提取事實性宣稱，忽略主觀評論或建議。"""

    CLAIM_VERIFY_PROMPT = """你是一個事實查核助理。請判斷以下宣稱是否能被原始證據支持。

宣稱：{claim}

原始證據：
{evidence_text}

請回答：
- VERIFIED：證據直接支持此宣稱
- PARTIAL：證據部分支持，但有出入（說明差異）
- UNVERIFIED：證據中找不到支持此宣稱的內容
- CONTRADICTED：證據與此宣稱矛盾（說明矛盾點）

回答格式：[狀態] [說明]"""

    def __init__(self, llm_client):
        self.llm = llm_client

    async def verify(
        self,
        answer_draft: AnswerDraft,
        governance_report: GovernanceReport
    ) -> CoVeResult:
        # Step 1: 提取事實宣稱
        claims = await self._extract_claims(answer_draft.summary)

        if not claims:
            return CoVeResult([], 1.0, [], 0.0)

        # Step 2: 組合所有證據文本
        evidence_text = self._compile_evidence(governance_report)

        # Step 3: 逐一驗證（可並行）
        verification_tasks = [
            self._verify_single_claim(claim, evidence_text)
            for claim in claims
        ]
        verified_claims = await asyncio.gather(*verification_tasks)

        # Step 4: 計算驗證率與信心分調整
        verified_count = sum(1 for c in verified_claims if c.is_verified)
        total = len(verified_claims)
        verified_ratio = verified_count / total if total > 0 else 1.0

        unverified = [c.claim_text for c in verified_claims if not c.is_verified]

        # 信心分調整邏輯：
        # - 100% 驗證通過 → +0.05
        # - 80%+ 驗證通過 → 不調整
        # - 60-80% → -0.10
        # - <60% → -0.20（觸發重新生成）
        if verified_ratio >= 1.0:
            adjustment = 0.05
        elif verified_ratio >= 0.8:
            adjustment = 0.0
        elif verified_ratio >= 0.6:
            adjustment = -0.10
        else:
            adjustment = -0.20

        return CoVeResult(verified_claims, verified_ratio, unverified, adjustment)

    async def _extract_claims(self, text: str) -> list[str]:
        """用 LLM 從回答中提取事實宣稱。"""
        prompt = self.CLAIM_EXTRACTION_PROMPT.format(answer_text=text)
        response = await self.llm.complete(prompt)
        return [line.strip() for line in response.split("\n") if line.strip()]

    async def _verify_single_claim(self, claim: str, evidence: str) -> VerificationClaim:
        """驗證單一宣稱。"""
        prompt = self.CLAIM_VERIFY_PROMPT.format(claim=claim, evidence_text=evidence)
        response = await self.llm.complete(prompt)

        is_verified = response.startswith("VERIFIED")
        return VerificationClaim(
            claim_text=claim,
            is_verified=is_verified,
            verification_note=response
        )
```

**新增檔案：** `src/llm_stock_system/core/sentiment_scorer.py`

```python
from dataclasses import dataclass
from datetime import date, timedelta
from enum import Enum

class SentimentLevel(Enum):
    EXTREME_FEAR = "極度恐慌"      # 0-20
    FEAR = "恐慌"                   # 20-40
    NEUTRAL = "中性"                # 40-60
    GREED = "貪婪"                  # 60-80
    EXTREME_GREED = "極度貪婪"      # 80-100

@dataclass
class SentimentScore:
    """個股或市場的情緒量化指標。"""
    score: int                         # 0-100，0=極度恐慌，100=極度貪婪
    level: SentimentLevel
    components: dict[str, float]       # 各分項分數
    trend: str                         # "上升" / "下降" / "持平"
    description: str                   # 人話描述

@dataclass
class SentimentComponent:
    """單一情緒分項。"""
    name: str
    score: float        # 0-100
    weight: float       # 加權比重
    data_points: int    # 使用了幾筆資料

class SentimentScorer:
    """多維度情緒量化計算器。

    計算公式：
    sentiment_score = Σ (component_score × weight)

    五個分項（靈感來自 CNN Fear & Greed Index，但針對個股調整）：
    1. 新聞情緒（News Sentiment）     - 權重 30%
    2. 價格動能（Price Momentum）      - 權重 25%
    3. 成交量異常（Volume Anomaly）    - 權重 20%
    4. 波動度（Volatility）            - 權重 15%
    5. 分析師語調（Analyst Tone）      - 權重 10%
    """

    WEIGHTS = {
        "news_sentiment": 0.30,
        "price_momentum": 0.25,
        "volume_anomaly": 0.20,
        "volatility": 0.15,
        "analyst_tone": 0.10,
    }

    # 沿用並擴展 forecast.py 的 token 列表
    BULLISH_TOKENS = (
        "看多", "看漲", "偏多", "上漲", "站上", "突破", "多方", "利多", "上修",
        "創高", "強勢", "買超", "加碼", "樂觀", "回升", "反彈",
    )
    BEARISH_TOKENS = (
        "看空", "看跌", "偏空", "下跌", "跌破", "失守", "空方", "利空", "下修",
        "破底", "弱勢", "賣超", "減碼", "悲觀", "衰退", "崩跌",
    )

    def calculate(
        self,
        governance_report: GovernanceReport,
        price_data: list[dict] | None = None,
        volume_data: list[dict] | None = None,
    ) -> SentimentScore:
        """計算綜合情緒指數。"""

        components = {}

        # 1. 新聞情緒（從 evidence 中的新聞類文件計算）
        components["news_sentiment"] = self._score_news_sentiment(governance_report)

        # 2. 價格動能（近期漲跌幅）
        components["price_momentum"] = self._score_price_momentum(price_data)

        # 3. 成交量異常（相對於均量的偏離）
        components["volume_anomaly"] = self._score_volume_anomaly(volume_data)

        # 4. 波動度（高波動 = 恐慌）
        components["volatility"] = self._score_volatility(price_data)

        # 5. 分析師語調
        components["analyst_tone"] = self._score_analyst_tone(governance_report)

        # 加權總分
        total = sum(
            components[k].score * self.WEIGHTS[k]
            for k in self.WEIGHTS
            if k in components
        )
        score = int(max(0, min(100, total)))

        # 判定等級
        level = self._score_to_level(score)

        # 趨勢（比較近 7 天 vs 近 30 天的新聞情緒）
        trend = self._calculate_trend(governance_report)

        # 生成人話描述
        description = self._generate_description(score, level, components)

        return SentimentScore(
            score=score,
            level=level,
            components={k: v.score for k, v in components.items()},
            trend=trend,
            description=description,
        )

    def _score_news_sentiment(self, report: GovernanceReport) -> SentimentComponent:
        """新聞情緒分數。

        邏輯：統計所有 NEWS 類證據中的多空 token 比例。
        100 = 全部正面，50 = 中性，0 = 全部負面。
        """
        news_evidence = [e for e in report.evidence if e.facet == DataFacet.NEWS]
        if not news_evidence:
            return SentimentComponent("news_sentiment", 50.0, self.WEIGHTS["news_sentiment"], 0)

        bull_count = 0
        bear_count = 0
        for ev in news_evidence:
            text = (ev.title or "") + " " + (ev.excerpt or "")
            text_lower = text.lower()
            bull_count += sum(1 for t in self.BULLISH_TOKENS if t in text_lower)
            bear_count += sum(1 for t in self.BEARISH_TOKENS if t in text_lower)

        total_signals = bull_count + bear_count
        if total_signals == 0:
            score = 50.0
        else:
            score = (bull_count / total_signals) * 100

        return SentimentComponent("news_sentiment", score, self.WEIGHTS["news_sentiment"], len(news_evidence))

    def _score_price_momentum(self, price_data: list[dict] | None) -> SentimentComponent:
        """價格動能分數。

        邏輯：
        - 計算 5 日、20 日、60 日報酬率
        - 正報酬 → 偏貪婪，負報酬 → 偏恐慌
        - 三個時間框架加權平均（近期權重高）
        """
        if not price_data or len(price_data) < 5:
            return SentimentComponent("price_momentum", 50.0, self.WEIGHTS["price_momentum"], 0)

        prices = [d["close"] for d in sorted(price_data, key=lambda x: x["date"])]

        def return_to_score(ret: float) -> float:
            """將報酬率轉換為 0-100 分。±10% 為滿分/零分。"""
            return max(0, min(100, 50 + (ret / 0.10) * 50))

        ret_5d = (prices[-1] / prices[-5] - 1) if len(prices) >= 5 else 0
        ret_20d = (prices[-1] / prices[-20] - 1) if len(prices) >= 20 else ret_5d
        ret_60d = (prices[-1] / prices[-60] - 1) if len(prices) >= 60 else ret_20d

        # 近期權重較高
        score = (
            return_to_score(ret_5d) * 0.50 +
            return_to_score(ret_20d) * 0.30 +
            return_to_score(ret_60d) * 0.20
        )

        return SentimentComponent("price_momentum", score, self.WEIGHTS["price_momentum"], len(prices))

    def _score_volume_anomaly(self, volume_data: list[dict] | None) -> SentimentComponent:
        """成交量異常分數。

        邏輯：今日量 vs 20 日均量的偏離程度。
        異常放大（> 2x 均量）→ 極端情緒（可能是恐慌或貪婪，需結合價格方向判斷）。
        """
        if not volume_data or len(volume_data) < 20:
            return SentimentComponent("volume_anomaly", 50.0, self.WEIGHTS["volume_anomaly"], 0)

        volumes = [d["volume"] for d in sorted(volume_data, key=lambda x: x["date"])]
        avg_20 = sum(volumes[-20:]) / 20
        latest = volumes[-1]

        if avg_20 == 0:
            return SentimentComponent("volume_anomaly", 50.0, self.WEIGHTS["volume_anomaly"], 0)

        ratio = latest / avg_20

        # 量價配合判斷需要價格方向，這裡簡化為：
        # ratio < 0.5 → 40（冷淡）
        # ratio 0.5-1.5 → 50（正常）
        # ratio > 1.5 → 偏離中性（需結合上下文）
        if ratio < 0.5:
            score = 40.0
        elif ratio <= 1.5:
            score = 50.0
        else:
            score = min(100, 50 + (ratio - 1.5) * 30)  # 放量偏貪婪

        return SentimentComponent("volume_anomaly", score, self.WEIGHTS["volume_anomaly"], len(volumes))

    def _score_volatility(self, price_data: list[dict] | None) -> SentimentComponent:
        """波動度分數。高波動 = 低分（恐慌），低波動 = 高分（穩定）。"""
        if not price_data or len(price_data) < 20:
            return SentimentComponent("volatility", 50.0, self.WEIGHTS["volatility"], 0)

        prices = [d["close"] for d in sorted(price_data, key=lambda x: x["date"])]
        returns = [(prices[i] / prices[i-1] - 1) for i in range(1, len(prices))]

        if not returns:
            return SentimentComponent("volatility", 50.0, self.WEIGHTS["volatility"], 0)

        import statistics
        vol_20d = statistics.stdev(returns[-20:]) if len(returns) >= 20 else statistics.stdev(returns)

        # 年化波動度：日波動度 × √252
        annual_vol = vol_20d * (252 ** 0.5)

        # 台股個股常見波動度範圍：15%-60%
        # 15% → 分數 80（穩定）
        # 35% → 分數 50（普通）
        # 60%+ → 分數 20（恐慌）
        score = max(0, min(100, 100 - (annual_vol - 0.15) / 0.45 * 80))

        return SentimentComponent("volatility", score, self.WEIGHTS["volatility"], len(returns))

    def _score_analyst_tone(self, report: GovernanceReport) -> SentimentComponent:
        """分析師語調分數。僅使用 HIGH tier 來源中的多空信號。"""
        high_trust = [e for e in report.evidence if e.source_tier == SourceTier.HIGH]
        if not high_trust:
            return SentimentComponent("analyst_tone", 50.0, self.WEIGHTS["analyst_tone"], 0)

        bull = 0
        bear = 0
        for ev in high_trust:
            text = (ev.title or "") + " " + (ev.excerpt or "")
            bull += sum(1 for t in self.BULLISH_TOKENS if t in text)
            bear += sum(1 for t in self.BEARISH_TOKENS if t in text)

        total = bull + bear
        score = (bull / total * 100) if total > 0 else 50.0

        return SentimentComponent("analyst_tone", score, self.WEIGHTS["analyst_tone"], len(high_trust))

    @staticmethod
    def _score_to_level(score: int) -> SentimentLevel:
        if score < 20:
            return SentimentLevel.EXTREME_FEAR
        elif score < 40:
            return SentimentLevel.FEAR
        elif score < 60:
            return SentimentLevel.NEUTRAL
        elif score < 80:
            return SentimentLevel.GREED
        else:
            return SentimentLevel.EXTREME_GREED

    def _calculate_trend(self, report: GovernanceReport) -> str:
        """比較近 7 天 vs 近 30 天的新聞情緒趨勢。"""
        # 依據證據的時間戳分組計算，回傳 "上升" / "下降" / "持平"
        # 簡化實作：比較前後兩段時間的多空比
        return "持平"  # TODO: 根據 evidence 的 published_date 分段計算

    def _generate_description(self, score, level, components) -> str:
        """生成人話情緒描述。"""
        desc = f"情緒指數 {score}/100（{level.value}）。"

        # 找出最極端的分項
        sorted_components = sorted(components.items(), key=lambda x: abs(x[1].score - 50), reverse=True)
        if sorted_components:
            top = sorted_components[0]
            name_map = {
                "news_sentiment": "新聞情緒",
                "price_momentum": "價格動能",
                "volume_anomaly": "成交量變化",
                "volatility": "波動度",
                "analyst_tone": "分析師觀點",
            }
            desc += f"主要驅動因素：{name_map.get(top[0], top[0])}（{top[1].score:.0f}/100）。"

        return desc
```

## 後續規劃：question_type 職責收斂與 Policy 架構升級

> 以下章節記錄經過 Codex AI 建議與人工審查後，確認要執行的架構調整方向。
> 分為「立刻執行」「下一輪執行」「記下暫緩」三個層級。

### 問題背景

`question_type` 目前同時負責 4 件事：

1. `InputLayer` 用它推導 intent、time range、forecast 特例與 fallback tags（`input_layer.py` line 641, 808）
2. `StructuredQuery` model validator 用它自動補 intent、facets、fallback tags（`models.py` line 258）
3. 少數 helper 用它做 legacy subtype 判斷（`llm.py` line 113、`fundamental_valuation.py` line 49）
4. `PolicyRegistry` 的 `QueryPolicy` 仍以 `question_type` 命名（`query_policy.py` line 34）

目標不是消滅 `question_type`，而是讓它退出主流程決策，只保留為 debug label。主流程的輸入合約應收斂為 `intent + controlled_tags + semantic_flags`，路由結果由 `ResolvedPolicy` 決定。

### 規劃項目 A（立刻執行）：`resolve_by_tags()` 只吃 `controlled_tags`

**優先級：** 最高 — 與剛完成的標籤擴充直接相關

**現狀問題：** 目前 `topic_tags` 是 `controlled_tags + free_keywords + fallback_tags` 的聯集（`models.py` line 282）。`PolicyRegistry.resolve_by_tags()` 直接吃這個混合欄位做路由，導致路由精準度隨 free_keywords 的增加而下降。在關鍵字從 95 擴充到 419 之後，這個問題會更嚴重。

**改動方向：**

- `models.py` 的 StructuredQuery validator：保留 `topic_tags` 作為給 generation/display 用的完整集合，但新增獨立的 `controlled_tags` 欄位（型別 `list[TopicTag]`）只傳遞 enum 值
- `query_policy.py` 的 `resolve_by_tags()`：改為只讀 `controlled_tags`，不讀混合後的 `topic_tags`
- `free_keywords` 只供 Retrieval Layer 做搜尋展開用，不參與路由

**影響檔案：**

| 檔案 | 改動 |
|---|---|
| `core/models.py` | StructuredQuery 新增 `controlled_tags: list[TopicTag]` 欄位 |
| `core/query_policy.py` | `resolve_by_tags()` 改讀 `controlled_tags` |
| `layers/input_layer.py` | `parse()` 回傳時確保 `controlled_tags` 獨立設值 |

### 規劃項目 B（立刻執行）：補 `policy_match_type` 可觀測欄位

**優先級：** 最高 — 成本低、價值高

**現狀問題：** 系統完全不知道「這次路由是精準命中還是 fallback 猜的」。尤其新增 13 個分類後，很多查詢可能只會 partial match，下游的 Validation Layer 無法據此區分對待。

**改動方向：**

- `QueryPolicy` 新增 `match_type` 欄位，至少區分 `exact`（intent + tags 完全命中）、`partial`（intent 命中但 tags 只部分符合）、`generic`（僅 intent 命中，無 tag 資訊）、`fallback`（全部退回預設）
- `resolve_by_tags()` 在解析過程中設定 `match_type`
- `ValidationLayer` 根據 `match_type` 調整信心分：`fallback` 時降低上限，`exact` 時允許更高信心

**影響檔案：**

| 檔案 | 改動 |
|---|---|
| `core/query_policy.py` | `QueryPolicy` 新增 `match_type` 欄位；`resolve_by_tags()` 設定值 |
| `layers/validation_layer.py` | 根據 `match_type` 微調信心分上限 |

### 規劃項目 C（下一輪執行）：時間窗口政策從 InputLayer 搬到 Policy 層

**優先級：** 中 — 影響面可控但非緊急

**現狀問題：** `InputLayer._detect_time_range()`（line 828）根據 `question_type` 硬編碼預設時間窗口（例如 `earnings_summary` → 最新一季、`market_summary` → 7 天）。這是「策略」而不是「解析」，概念上不屬於 InputLayer。

**改動方向：**

- 在 `QueryPolicy` 新增 `default_time_range_days: int | None` 與 `default_time_range_label: str | None` 欄位
- `InputLayer._detect_time_range()` 先檢查使用者是否明確指定時間範圍；若無，則由 resolved policy 的預設值填入，而非查 question_type 對照表
- 原有的 question_type → time_range 對照表保留為 fallback，待所有 policy 補齊後移除

**影響檔案：**

| 檔案 | 改動 |
|---|---|
| `core/query_policy.py` | `QueryPolicy` 新增時間窗口相關欄位 |
| `layers/input_layer.py` | `_detect_time_range()` 改為讀 policy 預設值 |
| `orchestrator/pipeline.py` | 調整呼叫順序，確保 policy 在 time_range 之前解析完成 |

### 規劃項目 D（下一輪執行）：清除殘留的 question_type 直接依賴

**優先級：** 中 — 順手清理

**現狀問題：** `llm.py`（line 113）和 `fundamental_valuation.py`（line 49）仍直接讀取 `question_type` 做判斷，繞過了 intent + tags 體系。

**改動方向：**

- 將這兩處改為讀取 `intent` + `controlled_tags` + resolved policy 的欄位
- 如果邏輯簡單到只是 if/elif 分支，可以直接用 intent enum 替代

### 規劃項目 E（記下暫緩）：P0 Document 端 tag schema 一致性

**暫緩理由：** query 端的 `topic_tags` 和 document 端 DB 存的 `topic_tags` 語意不同（前者是路由信號，後者是粗粒度 `Topic` 值）。但目前 retrieval 主要靠 `ticker + days_back` 做向量過濾，尚未走到 tag-aware hybrid retrieval。在真正需要 semantic tag matching 之前，改 DB schema 的投資報酬率太低。

**觸發條件：** 當 P0 的 Hybrid Retrieval 開始需要用 tag 做精準過濾時再啟動。屆時建議新增 `semantic_tags` / `route_tags` 獨立欄位，不再用 `topic_tags` 混兩種概念。

### 規劃項目 F（記下暫緩）：P3 CoVe 精細化配置

**暫緩理由：** P3 尚未啟動。Codex AI 建議的 `cove_mode`、`numeric_claim_expected`、`min_evidence_count`、`allow_match_types`、`max_claims_to_verify` 等欄位方向正確，但在功能不存在的階段預先定義配置結構，容易設計出與實際需求不合的介面。

**觸發條件：** P3 CoVe 開始實作時再定義。屆時 gating 條件應建立在「resolved policy + evidence quality + match quality」之上，而非僅依賴 intent 大類或 `cove_eligible: bool`。

### 規劃項目 G（記下暫緩）：question_type 重命名為 policy_id

**暫緩理由：** 語意上正確，但目前改名會牽動整個 test suite、helper script 和文件範例。在功能快速迭代的階段，rename 的維護成本高於收益。

**觸發條件：** 等 A~D 完成、架構穩定後，做一次性的全域 rename。`question_type` 在此之前保留為程式碼中的現有欄位名，不加 deprecated 標記（目前只有一個開發者，deprecation warning 無受眾）。

### 規劃執行順序總覽

```
立刻執行（與標籤擴充同一開發週期）
├── A. resolve_by_tags() 只吃 controlled_tags
└── B. 補 policy_match_type 欄位

下一輪執行
├── C. 時間窗口政策搬到 Policy 層
└── D. 清除 llm.py / fundamental_valuation.py 的 question_type 依賴

記下暫緩（各有明確觸發條件）
├── E. Document 端 tag schema 一致性 → 等 P0 Hybrid Retrieval 需要 tag 過濾時
├── F. P3 CoVe 精細化配置 → 等 P3 啟動時
└── G. question_type → policy_id rename → 等架構穩定後
```
